## 📌 О проекте

ВНИМАНИЕ: разработано с использованием ИИ. Используйте на свой страх и риск.

**Cheburproxy** — это быстрый, надежный и гибкий инструмент для прозрачной и явной маршрутизации трафика, написанный на Rust. Поддерживает работу как в режиме обычного SOCKS5 прокси, так и в полностью прозрачном режиме (TPROXY) на Linux.

Проект идеально подходит для сложной маршрутизации сетевого трафика: по доменам, базам GeoIP/GeoSite, а также с использованием автоматической проверки доступности ресурсов (Availability Check).

---

## ✨ Ключевые возможности

- 🚀 **Клиент и сервер** в одном проекте
- 🔗 **SOCKS5** (полноценная поддержка TCP и UDP)
- 👻 **Transparent Proxy (TPROXY)** для Linux — прозрачный перехват трафика без настройки приложений
- 🗺️ **Умная маршрутизация** по:
  - Заданным доменным именам
  - GeoIP (маршрутизация по странам)
  - GeoSite (категории сайтов)
- 🏷️ **Теги маршрутизации**:
  - `direct` — прямое соединение через провайдера (ISP)
  - `default` — маршрут по умолчанию (через заданный прокси)
  - _и любые ваши кастомные теги (например, `kz`, `eu`)_
- 🩺 **Availability Check (проверка доступности)** — умный автоматический фоллбэк на прокси, если ресурс недоступен напрямую
- 🛡️ **DNS over HTTPS (DoH)** — встроенный клиент (8.8.8.8, 1.1.1.1) с кэшированием
- ⚡ **Горячая конфигурация** — правила маршрутизации (`router.json`) обновляются "на лету" без перезапуска процесса (systemctl reload)!

## 🏗️ Архитектура

Cheburproxy может работать в нескольких вариантах развертывания в зависимости от ваших нужд.

### Основные режимы работы:

#### 1. Прокси‑сервер (Upstream Server)
- Принимает входящие SOCKS5 соединения от клиентов (включает поддержку **TCP и UDP**).
- Проксирует трафик без сложной логики маршрутизации (работает как скоростной транспортный узел).

#### 2. Прокси‑клиент: SOCKS5 Mode
- Локальный сервис принимает SOCKS5-подключения от ваших браузеров, мессенджеров или операционной системы.
- Выполняет интеллектуальный анализ трафика и маршрутизирует его по заданным правилам (GeoIP/GeoSite/Домены).
- Совместим с другими SOCKS5 прокси-серверами (xray, shadowsocks и т.д.).

#### 3. Прокси‑клиент: Transparent Mode (TPROXY)
- Тесно интегрируется с модулем ядра Linux (`TPROXY`).
- Незаметно перехватывает весь исходящий TCP/UDP трафик устройства.
- Приложения (в том числе консольные утилиты и игры) работают прозрачно и не требуют индивидуальных настроек прокси.

## 🧠 Логика маршрутизации трафика

Процесс принятия решения:

1. **Поиск домена / IP** в загруженных правилах `router.json`.
2. Если правило найдено — трафик автоматически направляется через маршрут с соответствующим тегом (`tag`).
3. Если правило **НЕ найдено**:
   - Активируется **Availability Check**.
   - Если целевой ресурс недоступен через прямое соединение (таймаут или блок), трафик заворачивается в прокси по умолчанию (`default`).
   - Если ресурс доступен напрямую, используется `direct`.
4. _Важно:_ Если проверка доступности отключена и домен отсутствует в правилах, то весь нераспознанный трафик идет через `default`.

## 🚀 Сборка из исходников

Для сборки проекта убедитесь, что у вас установлен инструментарий **Rust** и пакетный менеджер **Cargo**.

```bash
# Клонирование репозитория
git clone https://github.com/igroykt/cheburproxy.git
cd cheburproxy/src

# Сборка Release-бинарника
cargo build --release

# Бинарные файлы готовы! Ищите их в: target/release/
```

## 💻 Быстрый старт

### Запуск клиента
Для запуска локального клиента (маршрутизатора):
```bash
./target/release/cheburproxy -c /path/to/config.toml
```

### Запуск сервера
Для запуска на удаленной ноде (upstream-сервер):
```bash
./target/release/cheburproxy-server -c /path/to/config.toml
```

> **Примечание:** Один и тот же структурный файл `config.toml` может использоваться как для конфигурации клиента, так и сервера.

## ⚙️ Конфигурация

Проект опирается на два ключевых файла настройки:
- `config.toml` — базовые параметры сервиса (ip-адреса, порты, таймауты).
- `router.json` — гибко настраиваемые списки правил маршрутизации (поддерживает горячую перезагрузку измененного файла). Любые изменения в полях geoip/geosite требуют перезапуска (systemctl restart).

<details>
<summary><b>Посмотреть пример <code>config.toml</code></b></summary>

```toml
[general]
# IPv4: слушать на всех интерфейсах
listen = "0.0.0.0"
port = 1080
mode = "transparent" # или "proxy" для работы классическим SOCKS5 клиентом
username = ""
password = ""

[client]
# UDP desync
udp_desync_enabled = false
udp_desync_min_size = 64
udp_desync_max_size = 256

# Встроенный DNS proxy-сервер
dns_enabled = false
dns_listen_port = 53

# Таймауты и настройки кэша
upstream_proxy_timeout = 10
availability_check = true
availability_check_timeout = 5
dns_cache_ttl = 3600
availability_cache_ttl = 300

# Контексты сессий
context_enabled = true
context_ttl = 3600
```
</details>

## ⚠️ Ограничения и требования

* **ОС:** Поддерживается исключительно **Linux-based** системы (ввиду специфики TPROXY эвристик).
* **Права:** Для `Transparent mode (TPROXY)` необходимы повышенные привилегии (**root**).
* **Маршрутизация:** Требуется ручная настройка таблиц маршрутизации (`ip route`) и правил фаервола (`iptables`/`nftables`). Для удобства в проекте могут использоваться скрипты типа `start.sh`/`stop.sh`.
* **Внешние базы:** Для корректной работы GeoIP и GeoSite правил необходимо загрузить файлы `geoip.dat` и `geosite.dat` и поместить их в рабочую директорию программы.
